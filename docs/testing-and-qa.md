# Как проверить баги: тесты, чек-листы и риски

## 1. Что уже есть (запустить локально)

```bash
npm run lint          # ESLint — стиль и часть логических ошибок
npm run test          # Vitest — юнит/интеграция (server + client)
npm run test:coverage # Покрытие — видно непокрытые ветки
```

- **Сервер:** `tests/server/` — health, auth, projects, files, compile, collaborators, security, domain/validation.
- **Клиент:** `tests/client/` — LatexToolbar, useCustomSnippets, latexCompletions, **useEditor** (смена файла, selectProject, compile, refreshFiles, save, setMainFile, loadProjects), **useAuth** (login, logout, checkAuth).

Проверки: перед коммитом всегда `npm run lint` и `npm run test`. В CI это уже запускается. Сейчас ~103 теста (server + client).

---

## 2. Как ловить ошибки с отображением (контент при смене файла и т.п.)

Такие баги («переключил файл — контент не обновился») лучше всего ловить **юнит-тестами композаблов**:

1. **Мокать API** (`vi.mock('@/composables/useApi.js')`) и подменять ответы (разный `content` для разных файлов).
2. **Вызывать действие** (например `selectFile('other.tex')`) и **проверять реактивное состояние** (`editorContent.value` должно стать контентом нового файла).
3. **Учитывать побочные вызовы**: при смене файла может вызываться сохранение текущего (PUT), поэтому мок `api` должен обрабатывать и GET, и PUT, иначе тест «съест» не тот ответ.

Примеры: `tests/client/composables/useEditor.test.js` (смена файла, выбор проекта, компиляция, обновление списка файлов, сохранение, главный файл, загрузка проектов), `tests/client/composables/useAuth.test.js` (вход, выход, проверка сессии). Такой тест на `selectFile` упал бы до фикса бага с отображением при смене файла.

Для более тяжёлых сценариев (клик по дереву файлов в браузере) можно добавить **E2E** (Playwright/Cypress): открыть проект, кликнуть по второму файлу, проверить текст в редакторе.

## 3. Ручная проверка (чек-лист)

Пройди сценарии в браузере и отметь, что ломается.

### Авторизация и доступ
- [ ] Вход по логину/паролю (demo/demo или свои учётки)
- [ ] Выход, повторный вход
- [ ] Доступ без авторизации к `/`, `/login`, `/account`, редактору — редирект/ошибка по задумке
- [ ] Истечение сессии / невалидная кука — корректное поведение

### Проекты
- [ ] Список проектов в ЛК, создание нового, удаление
- [ ] Открытие проекта в редакторе (по клику / «Редактор»)
- [ ] Скачивание проекта (ZIP)
- [ ] Переименование проекта, смена main file

### Редактор и файлы
- [ ] Переключение между файлами — **содержимое редактора обновляется**
- [ ] Создание файла/папки, переименование, удаление
- [ ] Сохранение (авто и по кнопке), индикатор «Сохранено»
- [ ] Загрузка файлов (upload), загрузка ZIP
- [ ] Контекстное меню по ПКМ по файлу

### Компиляция и PDF
- [ ] Кнопка «Скомпилировать», лог и ошибки в UI
- [ ] Превью PDF после успешной компиляции
- [ ] Переход по ошибкам (клик по строке в логе → строка в редакторе)
- [ ] Авто-компиляция (вкл/выкл, интервал)
- [ ] Synctex: клик в PDF → строка в .tex (если реализовано)

### История и доп. функции
- [ ] История версий файла (если включена) — открытие, восстановление
- [ ] Поиск по проекту
- [ ] Определения (definitions), переход к определению
- [ ] Bib-ключи, автодополнение \cite

### Граничные случаи
- [ ] Очень большой файл / много файлов — не падает, не тормозит критично
- [ ] Невалидный LaTeX — компиляция с ошибками, сообщения в логе
- [ ] Одновременно два проекта в разных вкладках
- [ ] Сеть: обрыв при сохранении/компиляции — сообщение об ошибке, повтор

### Деплой и окружение
- [ ] Запуск через `npm start` (production build)
- [ ] Docker: сборка и запуск, health-check
- [ ] С Mongo/Redis (если используешь) — те же сценарии

Сохрани чек-лист (копия в Wiki/Notion) и проходи после больших изменений.

## 4. Где могут быть баги (по зонам)

| Зона | Что проверять |
|------|----------------|
| **Роутинг (Express)** | Порядок маршрутов (как с history), права доступа, 404/500 |
| **Файлы и пути** | path traversal, кодировка имён, бинарные файлы, размер |
| **Компиляция** | таймаут, убийство процесса, очистка временных файлов |
| **Редактор (Vue)** | реактивность при смене файла/проекта, утечки (таймеры, подписки) |
| **Realtime** | отключение при смене файла, конфликты правок |
| **Авторизация** | проверка владельца/доступа к проекту на каждом API |
| **Конфиг** | DATA_DIR, лимиты, Mongo/Redis URL при разных окружениях |

## 5. Как находить «скрытые» баги

- **Покрытие:** `npm run test:coverage` — смотри непокрытые ветки в `server/services/`, `server/routes/`, `src/composables/`.
- **Добавить тесты** на новый код и на баги, которые уже чинил (регрессия).
- **Логи сервера** при ручных проверках — смотреть на 4xx/5xx и странные запросы.
- **Консоль браузера** — ошибки JS, failed fetches.
- **Сеть (DevTools)** — какие запросы уходят при смене файла, компиляции, сохранении.

---

## 6. Краткий порядок проверки перед релизом

1. `npm run lint && npm run test`
2. Пройти чек-лист из раздела 2 (хотя бы критичные пункты)
3. Проверить один сценарий в production-like окружении (Docker или staging)
4. Зафиксировать известные ограничения/баги в README или Wiki

Если опишешь конкретный баг (шаги, ожидание, факт), можно добавить под него автотест в `tests/`.
