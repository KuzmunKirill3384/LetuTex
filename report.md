# Дизайн‑проект системы **LetuTex** – аналога Overleaf

## Цель и назначение

LetuTex предназначен для онлайн‑редактирования документов LaTeX в режиме реального времени. Пользователи смогут создавать, хранить и совместно редактировать проекты LaTeX через веб‑интерфейс, компилировать исходники в PDF, работать с системой контроля версий и обмениваться файлами. Платформа ориентирована на студентов, исследователей и преподавателей, которым требуется совместная работа над научными публикациями.

Основными задачами дизайна являются:

- **Организация совместной работы**. Несколько пользователей должны иметь возможность редактировать один документ и видеть изменения друг друга без блокировок. Конфликты изменений должны разрешаться автоматически.
- **Управление проектами и файлами**. Пользовательская структура папок, операции с файлами (создание, переименование, удаление), загрузка изображений, шаблоны документов.
- **Компиляция LaTeX**. Поддержка многих пакетов, быстрый возврат результата и изоляция процесса компиляции.
- **Интеграция с Git**. Возможность клонирования/выгрузки проектов через Git для работы в автономном режиме и сохранения истории изменений.
- **Масштабируемость и безопасность**. Архитектура должна поддерживать увеличение числа пользователей, защищать данные и быть удобной для развертывания в облаке.

## Основные требования

### Функциональные

1. **Регистрация и аутентификация** – создание и управление учётными записями, восстановление пароля, поддержка OAuth для интеграции с университетскими SSO.
2. **Создание и управление проектами** – список проектов пользователя, возможность предоставлять доступ другим пользователям (роль «владелец», «редактор», «читатель»), шаблоны для типовых документов.
3. **Редактор** – браузерный редактор с подсветкой синтаксиса LaTeX, дерево файлов, просмотр PDF. Редактор должен поддерживать совместное редактирование нескольких пользователей и отображать курсоры/выделения других участников.
4. **Система компиляции** – возможность компилировать проект в режиме полного (full) или инкрементального (incremental) синхронизированного режима. Используется TeX Live, запуск компиляции в изолированном контейнере, где проводится сборка (`latexmk`), а результаты и логи возвращаются пользователю.【47444287777994†L43-L49】
5. **Хранилище файлов** – хранение исходных .tex, изображений и бинарных файлов. Поддержка локального файлового хранилища или S3‑совместимых сервисов для масштабирования.
6. **Интеграция с Git** – отдельный сервис для работы с репозиторием (git‑bridge): клонирование, коммиты, получение diff и интеграция веток. Документы в системе и в репозитории должны синхронизироваться.
7. **API** – REST‑/GraphQL‑интерфейс для взаимодействия с клиентскими приложениями, мобильными клиентами и внешними сервисами.
8. **Журналирование и история** – хранение истории изменений, возможность отката к любой версии файла, просмотр diff.
9. **Управление конфигурацией проекта** – выбор набора пакетов TeX Live, управление зависимостями (например, BibTeX/BibLaTeX), настройка инструментов для проверки орфографии.

### Нефункциональные

- **Высокая доступность**: система должна выдерживать сотни одновременных соединений. Все компоненты должны легко масштабироваться горизонтально.
- **Безопасность**: поддержка HTTPS, шифрование данных на диске, контроль доступа по ролям, изоляция процесса компиляции (песочница).
- **Производительность**: время компиляции не должно превышать нескольких секунд; редактор должен реагировать мгновенно на действия пользователя.
- **Модульность**: сервисы должны быть независимы и запускаться отдельно, что упростит обновление и развертывание.
- **Расширяемость**: возможность добавлять новые форматы (Markdown), плагины и интеграции (например, Zotero).

## Архитектура системы

LetuTex будет построен на принципах микросервисов. Подход был выбран по аналогии с Overleaf Server Pro, который использует сервисы `sharelatex`, `git‑bridge`, `mongo` и `redis`【735150367404092†L118-L124】. Внутри контейнера Overleaf приложение разворачивается как набор микросервисов, которые запускаются под управлением `runit`【403603195486689†L140-L151】. Наше решение повторяет успешную модель, обеспечивающую гибкость и масштабируемость.

* **Browser (клиент)** – веб‑интерфейс, реализованный с использованием фреймворков React/Vue. Клиент открывает веб‑сокет для совместного редактирования и использует HTTP‑API для операций управления.
* **Web API** – основной сервис, обрабатывающий HTTP‑запросы, работу с пользователями, проектами, настройками, а также взаимодействие с другими сервисами. Реализован на Node.js (Express/Koa) или Go.
* **Real‑time service** – сервис веб‑сокетов. Отвечает за подключение клиентов, передачу операций редактирования между ними и сериализацию событий. Сервис реализует алгоритм синхронизации (OT или CRDT).
* **Document Updater** – сервис, обрабатывающий изменения документов: применяет операции к структуре документа, обеспечивает запись/чтение из Redis и MongoDB и периодически сбрасывает кэш в базу данных【403603195486689†L185-L255】.
* **MongoDB** – основная база данных для хранения документов, пользователей, настроек. Overleaf рекомендует использовать MongoDB и Redis【735150367404092†L118-L124】; это же решение применяем и здесь.
* **Redis** – in‑memory‑хранилище, где содержатся текущие версии документов для совместного редактирования. Документы периодически сбрасываются в MongoDB (пример: каждые 100 обновлений или через 5 минут после последней активности)【403603195486689†L223-L255】.
* **File Store** – сервис хранения бинарных файлов (изображений, таблиц). Можно использовать локальную файловую систему, S3‑совместимое хранилище или MinIO. Сервис отвечает за загрузку/выгрузку файлов и отдачу их в компилятор.
* **Compile Service (CLSI)** – сервис компиляции. При получении запроса от Web API загружает необходимые файлы, формирует директорию проекта и запускает Docker‑контейнер с установленным TeX Live. Каждая компиляция выполняется в отдельном контейнере, что обеспечивает изоляцию и воспроизводимость【47444287777994†L43-L49】. В Overleaf используется два режима компиляции — full и incremental, мы реализуем аналогичные процессы: при `full`‑режиме весь проект загружается из MongoDB, а при `incremental` обновляются только изменённые файлы, что ускоряет сборку【403603195486689†L277-L337】【403603195486689†L345-L399】.
* **Git Bridge (необязательный)** – сервис для интеграции с Git, позволяющий клонировать проекты, выполнять push/pull и синхронизировать историю. Он слушает порт 22 (или кастомный) и взаимодействует с Web API.

### Реализация совместного редактирования

Ключевой задачей является разрешение конфликтов между параллельными правками. Существует два подхода: *Operational Transformation* (OT) и *Conflict‑free Replicated Data Types* (CRDT).

- **Operational Transformation**. OT предусматривает, что у каждого клиента есть копия документа; пользователь редактирует локальную копию без блокировок, а изменения отправляются другим клиентам. При получении чужих изменений клиент трансформирует их с учётом уже выполненных у себя операций, чтобы сохранить согласованность【739047097872646†L166-L176】. OT поддерживает причинно‑следственное упорядочивание и конвергенцию, обеспечивая одинаковое состояние документа у всех участников【739047097872646†L166-L176】. Основной недостаток – сложность реализации для древовидных структур (например, HTML‑деревьев) и необходимость написания трансформаций для всех типов операций.

- **CRDT**. CRDT — структуры данных, реплицируемые на нескольких узлах, которые позволяют обновлять любую реплику независимо и конвергировать состояние без координации. Изменения на репликах разрешаются алгоритмом, который гарантирует сходимость и не требует блокировок; изменения конфликтов автоматически разрешаются, при этом реплики гарантированно сходятся【778176493840118†L135-L146】. CRDT‑подход проще в теории и обеспечивает конечную согласованность, но может требовать большей пропускной способности (особенно state‑based варианты) и более сложного планирования ресурсов.

Для LetuTex предлагаем использовать **OT** для простоты интеграции с существующими библиотеками (например, [ShareDB], Yjs или Ot.js). Wikipedia отмечает, что в OT система обычно реплицирует документ на каждом клиенте, клиенты работают с локальными копиями и преобразуют операции при получении изменений, что обеспечивает высокую отзывчивость в условиях высокой задержки сети【739047097872646†L166-L176】.

### Сервис компиляции

Сервис компиляции (CLSI) является отдельным процессом, который отвечает за сборку проектов. Основные моменты:

1. **Изолированное окружение**. Для каждого запроса создаётся новый Docker‑контейнер, содержащий установленный TeX Live и необходимые программы (`latexmk`, `pdflatex`, `bibtex`), что соответствует методике Overleaf【47444287777994†L43-L49】. Такой подход обеспечивает безопасность и гарантирует воспроизводимость сборок.
2. **Режимы компиляции**:
   - **Полный (full)**: Web API инициирует запрос на компиляцию, сервис Document Updater сбрасывает актуальные данные из Redis в MongoDB, затем CLSI загружает все файлы проекта, записывает их в рабочую директорию и запускает сборку. После завершения возвращаются PDF и логи. Процесс подробно описан в документации Overleaf【403603195486689†L277-L337】.
   - **Инкрементальный (incremental)**: при повторных сборках CLSI получает только изменённые документы из Redis, а статические файлы читаются из локального кэша. Если хэш состояния не совпадает, сервис автоматически переключается на полный режим【403603195486689†L345-L399】.
3. **Очистка ресурсов**. После завершения компиляции контейнер останавливается и удаляется (или очищается по расписанию), что предотвращает накопление мусора и утечку памяти.

### Хранилище данных

- **MongoDB**. Хранит пользователей, проекты, дерево файлов и историю изменений. Сильные стороны — гибкая схема и поддержка масштабирования по шардированию.
- **Redis**. Служит кэшем и временным хранилищем для активных документов и отдельных операций совместного редактирования. Логика Redis‑кеширования в Overleaf включает загрузку документа из MongoDB в Redis при открытии, запись обновлений, регулярный сброс в базу (через 5 минут или после 100 изменений) и финальный сброс после выхода последнего пользователя【403603195486689†L223-L255】.
- **Файловое хранилище / S3**. Сервис File Store хранит бинарные объекты. Использование S3‑совместимого хранилища позволяет масштабировать систему и организовать резервное копирование.

### Интеграция с Git

Сервис Git Bridge реализует протокол Git (SSH/HTTPS) и взаимодействует с Web API и MongoDB. Пользователи могут клонировать проекты, коммитить изменения и отправлять их обратно. В Overleaf этот сервис опционален【735150367404092†L118-L124】; для LetuTex он также может быть подключаемым модулем. Git‑Bridge синхронизирует состояние файловой системы и базы данных. Предусматривается автосохранение – при push на сервер Web API отправляет операции изменения в Document Updater, который обновляет Redis и MongoDB.

### Сетевая инфраструктура и деплой

Развёртывание LetuTex можно реализовать на контейнерной платформе (Docker Compose на этапе разработки и Kubernetes в производстве). В Overleaf контейнер `sharelatex` публикует только порт 80 для доступа к Nginx, а MongoDB (27017) и Redis (6379) доступны внутри сети Docker【735150367404092†L152-L159】. Аналогично, наши сервисы будут общаться внутри подсети через сервис‑дискавери, а внешний входной трафик будет поступать через Nginx/HAProxy.

**Состав серверов и сервисов**:

|-------|-------------|---------|
| **Web API** | Node.js/TypeScript (Express, NestJS) | REST/GraphQL API, авторизация, управление проектами |
| **Real‑time** | Node.js (socket.io) или Go (Gorilla WebSocket) | Обеспечивает WebSocket‑соединения, передачу операций OT |

### Выбор технологий

- **Frontend**: React с TypeScript или Vue.js. Для редактора используются готовые библиотеки (Monaco Editor или CodeMirror) с возможностью интеграции OT‑протокола. Просмотр PDF реализуется через PDF.js.
- **Backend**: Node.js/TypeScript удобен благодаря большому числу библиотек (socket.io, ShareDB). Однако для тяжёлых операций можно использовать Golang или Rust для повышения производительности.
- **Контейнеризация**: Docker используется для всех сервисов; Kubernetes обеспечивает автоматическое масштабирование и балансировку. Блог Overleaf отмечает, что переход к запуску микросервисов в Docker упростил работу команды и улучшил управляемость системы【47444287777994†L83-L92】. Возможное расширение – запуск кластера через Kubernetes, как упомянуто в статье (Overleaf тестирует оркестратор для управления многочисленными контейнерами)【47444287777994†L94-L99】.
- **Хранение данных**: MongoDB и Redis обеспечивают масштабируемое хранение и кэширование; для бинарных файлов – S3.
- **Алгоритм совместной работы**: выбираем библиотеку OT (например, Yjs с адаптером WebSocket) либо CRDT (Automerge) в зависимости от требований к производительности и интеграции. Документ должен хранить список операций и состояние.
- **Безопасность**: JWT/OAuth2, HTTPS, ограничение доступа к Redis/MongoDB, регулярные резервные копии, защита от инъекций.

### Потоки данных

Чтобы обеспечить согласованность между клиентами, сервисы обмениваются данными по следующей схеме:

1. Пользователь входит в систему через Web API, открывает проект и загружает структуру файлов. Клиент открывает WebSocket‑соединение с Real‑time service.
2. При открытии документа Real‑time запрашивает через Document Updater текущую версию из Redis. Если её нет, она загружается из MongoDB и помещается в Redis【403603195486689†L185-L255】.
3. Пользователь вносит изменения. Клиент отправляет операцию OT через WebSocket. Real‑time передаёт её другим участникам и в Document Updater. Document Updater обновляет состояние в Redis.
4. Каждые N операций или через заданный интервал Document Updater сбрасывает изменения в MongoDB【403603195486689†L223-L255】.
5. Пользователь инициирует компиляцию через Web API. Сервис передаёт запрос в Document Updater (при необходимости выполняется сброс Redis‑кеша) и затем отправляет запрос в Compile Service. CLSI формирует рабочую директорию, запускает контейнер, компилирует проект и возвращает PDF/лог【403603195486689†L277-L337】【403603195486689†L345-L399】.
6. При завершении сессии Real‑time проверяет наличие других клиентов и инициирует финальный сброс в MongoDB【403603195486689†L249-L255】.

## Дизайн интерфейса (UI/UX)

### Веб‑интерфейс

1. **Главная панель**: список проектов, кнопка «Создать проект», выбор шаблона. Возможность сортировки и фильтрации.
2. **Окно редактора**: разделено на три области:
   - *Дерево файлов* – слева отображается структура проекта. Контекстное меню для создания/переименования/удаления файлов.
   - *Редактор кода* – центральная часть, где расположен редактор с подсветкой, функциями автодополнения, подсказками по LaTeX. Видны курсоры других пользователей, выделения и комментарии.
   - *Предпросмотр PDF* – справа отображается скомпилированный документ. Обновляется автоматически после компиляции.
   - Внизу или справа – панель логов компиляции, где выводятся предупреждения и ошибки.
3. **Управление пользователями**: модальное окно для приглашения участников по электронной почте, назначение ролей (редактор/читатель).
4. **История**: вкладка или панель для просмотра списка коммитов, diff‑просмотр, возможность отката.

### Адаптивность

Интерфейс должен быть оптимизирован для различных экранов (ноутбуки, планшеты). Желательно обеспечить оффлайн‑режим – при временной потере связи редактор сохраняет изменения локально, а после восстановления соединения отправляет их на сервер.

## Планы развития и возможные улучшения

- **Отслеживание изменений и комментарии**: внедрить режим правок с возможностью принятия/отклонения, как в Overleaf Server Pro. Это требует расширения модели данных и интерфейса.
- **Интеграция с системами цитирования**: подключение менеджеров библиографий (Zotero, Mendeley) для автоматической подстановки ссылок.
- **Поддержка других форматов**: например, редактирование Markdown с последующей конвертацией в LaTeX через Pandoc.
- **Мобильное приложение**: разработать нативные приложения для iOS/Android, использующие существующий API.
- **Проверка грамматики и стилистики**: интеграция языковых моделей для помощи в написании текста.

## Заключение

Проект LetuTex следует лучшим практикам реализации платформ для совместного редактирования документов LaTeX. Использование микросервисной архитектуры, изолированной компиляции и алгоритмов Operational Transformation/CRDT позволит обеспечить надёжность, масштабируемость и удобство для конечных пользователей. При разработке важно соблюдать баланс между функциональностью и простотой интерфейса, следовать принципам безопасности и обеспечения высокой доступности.
